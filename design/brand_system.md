# Companion Health — Brand System
Version: v1.0  
Status: ENFORCED  

---

# 1. PURPOSE

This document defines how Companion Health is visually represented across:

- Reports
- Product UI
- API surfaces
- Demo environments
- Marketing assets

This is a strict system. Not a guideline.

---

# 2. BRAND ARCHITECTURE

Two primary identities:

## 2.1 Companion Health (Parent Layer)
Represents:
- Intelligence layer
- Platform
- System authority

Used for:
- Reports
- Platform UI
- Institutional presence

---

## 2.2 Fetch™ API (Product Layer)
Represents:
- API product
- Developer-facing interface
- Integration layer

Used for:
- API documentation
- Developer portal
- Integration surfaces

---

# 3. LOGO SYSTEM

## 3.1 Logo Types

You have 3 distinct assets:

### A. Brandmark (Symbol Only)
Use when:
- Space is constrained
- System UI environments
- Favicon / minimal identity

NEVER use in isolation in formal reports.

---

### B. Logotype (Full Logo)
Use when:
- Reports
- Landing pages
- Headers

Default representation of brand.

---

### C. Fetch Logo
Use when:
- API sections
- Developer documentation
- Integration surfaces

Must NOT replace Companion logo at system level.

---

# 4. LOGO USAGE RULES

## 4.1 Companion Health

### Use Full Logo:
- Report header
- Navigation bar
- Cover page

### Use Brandmark:
- UI elements
- System indicators
- Background watermark (low opacity)

---

## 4.2 Fetch™

Use only inside:
- API sections
- Endpoint documentation
- Developer areas

Never position Fetch above Companion in hierarchy.

---

## 4.3 Color Variants

Allowed:
- Companion Green
- White
- Tail Green
- Bark Blue (secondary)

Rules:

- Dark background → use White logo
- Light background → use Companion Green
- Avoid mixing logo colors randomly

---

## 4.4 Clear Space

Minimum spacing around logo:

= height of the “C” in Companion

No elements inside this boundary.

---

## 4.5 What NOT to do

- Do not stretch logos
- Do not add shadows
- Do not recolor outside brand palette
- Do not place on noisy backgrounds
- Do not rotate or animate logos

---

# 5. TYPOGRAPHY SYSTEM

STRICT — NO DEVIATION

---

## 5.1 Primary Fonts

### Faire Octave
Use for:
- Headlines
- Section titles
- Key statements

Tone:
Editorial authority

---

### Suisse Intl
Use for:
- Body text
- UI labels
- Data
- Tables

Tone:
Precision, clarity

---

### Suisse Intl Mono
Use for:
- API labels
- Grid references
- Code blocks
- System metadata

Tone:
Technical layer

---

## 5.2 Typography Hierarchy

### H1 (Hero)
- Faire Octave
- Large scale
- Tight line height

---

### H2 (Section Titles)
- Faire Octave
- Medium scale

---

### Body
- Suisse Intl
- Clean, readable

---

### Metadata / Labels
- Suisse Intl Mono
- Small caps or tight spacing

---

## 5.3 Rules

- No mixing additional fonts
- No fallback fonts unless system required
- No decorative typography
- No excessive weight variation

---

Follow brand guide strictly.

---

# 7. GRAPHIC SYSTEM

## 7.1 Background Elements

Use brand shapes ONLY:

- Radial fades
- Soft signal fields
- Low-opacity gradients

Purpose:

→ Suggest system activity  
NOT decoration  

---

## 7.2 Data Visualisation

Charts must be:

- minimal
- clean
- non-decorative

No:

- 3D charts
- heavy shadows
- unnecessary animation

---

## 7.3 Grid System

All layouts must follow:

- consistent column grid
- predictable spacing
- aligned edges

No free-form layouts.

---

# 8. REPORT DESIGN RULES

Reports (like Market Intelligence) must:

- feel editorial + institutional
- maintain density
- avoid product UI styling

---

## Reports SHOULD:

- use serif headlines (Faire Octave)
- use structured grids
- use restrained color

---

## Reports SHOULD NOT:

- behave like dashboards
- include interactive UI patterns
- simulate product screens

---

# 9. PRODUCT UI RULES

Product UI must:

- be system-first
- be minimal
- avoid editorial styling

---

## UI SHOULD:

- use Suisse Intl primarily
- reduce decoration
- emphasize data clarity

---

## UI SHOULD NOT:

- look like a report
- use heavy serif typography
- use large narrative blocks

---

# 10. CONSISTENCY ENFORCEMENT

Every new screen, page, or asset must pass:

1. Typography check
2. Logo usage check
3. Color check
4. Grid alignment check

If any fail → redesign required

---

# FINAL PRINCIPLE

Companion Health is not a brand that expresses.

It is a system that reveals.

Design must reflect that.
